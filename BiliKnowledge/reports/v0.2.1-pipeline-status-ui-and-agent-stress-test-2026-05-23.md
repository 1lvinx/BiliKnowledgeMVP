## 6. Feature Behavior

### Pipeline Status Card
- Displays on dashboard after the metrics grid
- Shows 6 metric cards: Videos, Pending, Notes, Projects, Reviewed, Last Updated
- Shows 5 pipeline flags as tag pills: Manifest, Notes, Projects, Index, Validated (green ✓ or red ✗)
- **Refresh** button re-fetches `processing_status.json` via Tauri command
- **Run Validation** button runs `validate_knowledge_base.py`, then auto-refreshes all data including pipeline status
- **Loading state**: Shows "Loading pipeline status..." text
- **Error state**: Shows error notice with message
- **Stale warning**: Shows notice if not all pipeline flags are true

### Data Flow
```
App mount → fetchProcessingStatus() → invoke("get_processing_status") → Rust reads manifest/processing_status.json → JSON string → React parses → render card
Run Validation → runPythonScript("validate_knowledge_base.py") → script runs → fetchVideos() + fetchProjects() + fetchProcessingStatus() → card updates
```

## 8. Bugs Found and Fixed

### Bug 1: MacTagPill tone type mismatch
- **Symptom**: `TS2322: Type '"positive"' is not assignable to type '"neutral" | "warm" | "cool" | "critical" | "success" | undefined'`
- **Root Cause**: Used `"positive"` tone which doesn't exist in MacTagPill component
- **Fix**: Changed to `"success"`
- **File**: `App.tsx`
- **Regression Risk**: None

### Bug 2: MacInlineNotice tone type mismatch
- **Symptom**: `TS2322: Type '"warning"' is not assignable to type '"neutral" | "success" | "error" | undefined'`
- **Root Cause**: Used `"warning"` tone which doesn't exist in MacInlineNotice component
- **Fix**: Changed to `"neutral"`
- **File**: `App.tsx`
- **Regression Risk**: None

## 13. Collaboration Mechanism Evaluation

### What worked
- **Role separation forced discipline.** Each agent had a specific checklist. The Bugfix Agent caught 2 type errors that would have broken the build.
- **Gate model prevented premature action.** Running all tests before claiming completion ensured no false claims.
- **Sprint orchestrator kept scope tight.** No scope creep, no unnecessary refactors.
- **Security check was not skipped.** Explicitly scanned for secrets and forbidden paths.

### What failed
- **No real multi-agent parallelism.** All roles executed sequentially by a single Claude instance.
- **Bugfix Agent was reactive, not proactive.** Type errors were found during build, not during code writing.

### What should become a reusable Skill
- `repo-audit` — git status + diff + secret scan
- `release-gate` — run all tests, output PASS/BLOCK matrix
- `pipeline-status-ui` — JSON status → card → refresh → validation

### What should be simplified next time
- For 3-file changes, 10 roles is heavy. Better for multi-file or cross-project tasks.
- The Product/UI Agent role could be merged with Frontend Engineer for small UI tasks.
